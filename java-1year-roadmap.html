Core Java Fundamentals

Month 2 Core Java Depth

Week 5 — Exception Handling

  • Exception hierarchy: Throwable → Error, Exception → RuntimeException
  • Checked exceptions (must declare/handle) vs unchecked (RuntimeException subtypes)
  • try-catch-finally; multiple catch blocks; multi-catch (|)
  • try-with-resources (AutoCloseable): automatic closing in reverse order
  • Custom exception classes: message, cause, custom fields
  • Exception propagation: checked exceptions must be declared with throws
  • Suppressed exceptions: body exception is primary; close exceptions become suppressed
  • Best practices: don't swallow exceptions; don't use exceptions for control flow; log with context

Practice File reader with try-with-resources; custom exception hierarchy for a domain (e.g., InsufficientFundsException)

Week 6 — Collections Framework

  • List: ArrayList (resizable array, O(1) random access, amortized O(1) append), LinkedList (doubly-linked, O(1) head/tail, poor cache locality), Vector (legacy, synchronized)
  • Set: HashSet (hashing, unordered), LinkedHashSet (insertion order), TreeSet (sorted, red-black tree, O(log n))
  • Map: HashMap (buckets, O(1) avg), LinkedHashMap (insertion/access order), TreeMap (sorted), Hashtable (legacy)
  • Queue/Deque: ArrayDeque (preferred), PriorityQueue (min-heap)
  • equals/hashCode contract: equal objects must have equal hash codes; never mutate keys after insertion
  • Comparable vs Comparator: natural order vs custom comparator
  • Collections utilities: sort, unmodifiableList, synchronizedList, frequency, disjoint
  • Immutable collections: List.of, Map.of, Set.of (Java 9+)

Practice Phone book (HashMap); top-K frequent words (PriorityQueue); anagram grouper (HashMap<String, List<String>>)

Week 7 — Generics + I/O

  • @SafeVarargs; generic array creation limitations
  • java.io: File, FileReader/Writer, BufferedReader/Writer, byte streams, PrintWriter
  • java.nio.file: Path, Paths.get, Files.readAllLines, Files.walk, Files.copy
  • Serialization: Serializable, serialVersionUID, ObjectOutputStream/InputStream; why Java serialization is problematic (prefer Jackson/Protobuf)

Week 12 — Modern Java 18–21 + JVM Architecture

  • Java 21 Virtual Threads: lightweight threads managed by JVM; Thread.ofVirtual().start(); Executors.newVirtualThreadPerTaskExecutor(); pinning on synchronized (use ReentrantLock); not for CPU-bound work
  • Structured concurrency: StructuredTaskScope; child tasks finish/cancel with parent scope; prevents orphaned tasks
  • Scoped values: ScopedValue — immutable, per-thread-hierarchy alternative to ThreadLocal
  • Sequenced collections: SequencedCollection, SequencedSet, SequencedMap — consistent getFirst/getLast/addFirst/addLast
  • JVM architecture: classloader hierarchy (Bootstrap → Platform → Application → custom); runtime data areas (Heap, Metaspace, Java Stacks, PC Register, Native Method Stack)
  • Class loading: loading → linking (verification, preparation, resolution) → initialization; <clinit> runs static initializers exactly once per classloader; parent-first delegation

Practice Demo: virtual thread HTTP fan-out (100K requests); structured concurrency scope with ShutdownOnFailure; measure throughput vs platform threads

Month 4 Concurrency + JVM Deep

Week 13 — Concurrency Fundamentals

  • Thread creation: extends Thread, implements Runnable, Callable + Future
  • Thread states: NEW → RUNNABLE → BLOCKED / WAITING / TIMED_WAITING → TERMINATED
  • Thread.sleep, join, interrupt, isInterrupted; daemon threads
  • synchronized keyword: instance method (locks this), static method (locks class), synchronized block (explicit monitor)
  • Intrinsic locks and monitors; wait/notify/notifyAll
  • Java Memory Model: visibility without synchronization is undefined; happens-before relation; read from stale cache possible without JMM anchors
  • volatile: visibility + ordering guarantee for that variable; does NOT make compound ops atomic

Practice Producer-consumer using wait/notify; demonstrate data race (two threads incrementing shared counter); fix with synchronized

Week 14 — Advanced Concurrency

  • ReentrantLock: lock/unlock (always in finally), tryLock(timeout), fairness parameter
  • ReadWriteLock: many concurrent readers OR one exclusive writer; ideal for read-heavy caches
  • StampedLock: optimistic read mode; lower overhead for short reads
  • Condition: await/signal/signalAll; multiple conditions per lock (e.g., notFull + notEmpty for bounded buffer)
  • Atomic variables: AtomicInteger, AtomicLong, AtomicReference, AtomicStampedReference, LongAdder (striped — faster under contention)
  • CAS: compare-and-swap hardware instruction; optimistic update loop; ABA problem and fix (AtomicStampedReference)
  • ThreadLocal: per-thread storage; must call remove() in thread pool finally blocks or memory leaks occur
  • Coordination primitives: CountDownLatch (one-time gate), CyclicBarrier (reusable barrier), Phaser (dynamic participants), Semaphore (N permits), Exchanger

Practice Bounded buffer with ReentrantLock + Condition; bank transfer deadlock → fix with consistent lock ordering; synchronized vs AtomicLong vs LongAdder benchmark

Week 15 — Thread Pools + CompletableFuture

  • ExecutorService: submit, invokeAll, invokeAny, shutdown, awaitTermination
  • ThreadPoolExecutor: corePoolSize, maximumPoolSize, keepAlive, BlockingQueue type, RejectedExecutionHandler (AbortPolicy, CallerRunsPolicy, DiscardPolicy)
  • Executors factory: newFixedThreadPool (unbounded queue — danger!), newCachedThreadPool, newSingleThreadExecutor
  • ScheduledExecutorService: schedule, scheduleAtFixedRate, scheduleWithFixedDelay
  • ForkJoinPool: work-stealing; RecursiveTask/RecursiveAction; common pool shared — beware blocking
  • CompletableFuture: supplyAsync/runAsync; transformations: thenApply (sync), thenApplyAsync; chaining: thenCompose; combining: thenCombine, allOf, anyOf
  • Exception handling: exceptionally, handle, whenComplete
  • Timeouts: orTimeout (Java 9+), completeOnTimeout

Practice Fan-out to 3 external services concurrently with timeout + fallback + metrics; custom executor with bounded queue and CallerRuns backpressure

Week 16 — GC + Performance Engineering

  • GC algorithms: Serial (single-thread, tiny heaps), Parallel (throughput, batch jobs), G1 (default Java 9+, balanced, region-based, bounded pauses), ZGC (sub-ms pauses, Java 17+), Shenandoah (concurrent compaction)
  • Heap dump analysis: trigger with jcmd <pid> GC.heap_dump or -XX:+HeapDumpOnOutOfMemoryError; open in MAT; dominator tree; retained size; path to GC root
  • JFR (Java Flight Recorder): jcmd <pid> JFR.start duration=60s; open in JMC (JDK Mission Control); CPU, allocation, GC, lock events
  • JMH benchmarking: @Benchmark, @Warmup, @Measurement, @Fork, Blackhole.consume() to prevent dead-code elimination
  • async-profiler: -e cpu for CPU flame graphs; -e alloc for allocation profiling; reads actual native stack (avoids safepoint bias)
  • Common memory leaks: static collections, classloader leaks, event listener not removed, unclosed resources, ThreadLocal not removed in pools

Microservices Architecture + Docker + Kubernetes

Month 8 Microservices Architecture

Week 29 — Microservices Foundations + DDD

  • Monolith vs microservices: monolith = simpler ops, faster dev, harder to scale independently; microservices = independent deploy/scale, team autonomy, higher ops complexity
  • When NOT to use microservices: small team, greenfield (unknown boundaries), tight latency budgets
  • Domain-Driven Design basics: bounded context (clear boundary of a domain model), aggregate (cluster of entities with one root), domain events, ubiquitous language
  • Service boundary heuristics: split by business capability, by data ownership (each service owns its DB), by change frequency, by team size (Conway's Law)
  • Hexagonal architecture (Ports & Adapters): domain core (pure business logic) + ports (interfaces) + adapters (HTTP, DB, Kafka, CLI); domain never depends on frameworks
  • Clean architecture: entities → use cases → interface adapters → frameworks; inner circles never import outer

Practice Split Phase 3 e-commerce project into 3 bounded contexts: User, Product Catalog, Order Fulfillment. Define APIs and events between them. Draw context map.

Week 30 — Inter-Service Communication + Resilience

  • Synchronous REST: WebClient (reactive, non-blocking, preferred); OpenFeign (declarative client with @FeignClient); RestTemplate (legacy, deprecated)
  • Asynchronous: event-driven via Kafka; services decoupled, higher availability, eventual consistency
  • API Gateway: Spring Cloud Gateway; routing by path/host; rate limiting (RequestRateLimiter filter); auth offloading; request/response transformation
  • Service discovery: Eureka (client-side load balancing); in Kubernetes, use K8s DNS instead
  • Resilience4j: @CircuitBreaker (CLOSED/OPEN/HALF_OPEN); @Retry (with backoff + jitter); @RateLimiter; @Bulkhead (limit concurrent calls); @TimeLimiter (timeout)
  • Fallback methods: return cached data or degraded response when circuit is open

Practice Order service calls Inventory service via Feign; add Resilience4j circuit breaker + retry with exponential backoff; simulate Inventory service failure; verify fallback

Week 31 — Distributed Data Patterns

  • 2PC problem: two-phase commit across services requires both to be available simultaneously — fragile, blocking; avoid in microservices
  • Saga pattern: choreography (services react to events, no central coordinator — eventual consistency, hard to debug) vs orchestration (saga orchestrator coordinates steps, easier to reason about)
  • Outbox pattern: write business data AND outbox event in same DB transaction; separate publisher reads outbox and sends to Kafka; guarantees atomicity without 2PC. CDC via Debezium as alternative to polling
  • CQRS: Command Query Responsibility Segregation — write model (domain events, aggregates) ≠ read model (denormalized projections); enables independent scaling
  • Event sourcing: store state as sequence of events; rebuild by replaying; audit trail built in; complex to query
  • Idempotency: idempotency key (UUID per operation); deduplication store (Redis or DB); upsert semantics; critical for retries
  • Distributed tracing: OpenTelemetry SDK; trace propagation via W3C traceparent header; spans for each service; Jaeger/Zipkin for visualization

Practice Implement Outbox pattern: order service writes order + outbox event in one transaction; outbox poller publishes to Kafka; inventory service consumes idempotently

Week 32 — Docker

  • Images and containers: image = read-only layers; container = running instance of image
  • Dockerfile: FROM, WORKDIR, COPY, RUN, EXPOSE, ENTRYPOINT/CMD
  • Multi-stage builds: Stage 1: JDK + build; Stage 2: slim JRE + copy artifact only. Reduce image size from ~500 MB to ~100 MB
  • Volumes: named volumes (persistent data), bind mounts (dev: live code reload)
  • Networking: bridge (default, container-to-container), host, overlay (Swarm/K8s); DNS resolution by container name
  • docker-compose: define multi-service local environment; depends_on; health checks; environment variables
  • Security: run as non-root user; never store secrets in layers; use .dockerignore; scan images (docker scout, Trivy)

Practice Dockerize all 3 microservices (multi-stage); write docker-compose.yml with Postgres, Redis, Kafka (Redpanda); verify startup order with health checks

Month 9 Kubernetes

Week 33 — Kubernetes Core Objects

  • Pod: smallest deployable unit; one or more containers sharing network namespace and storage
  • Deployment: manages ReplicaSets; declarative rolling updates; rollback with kubectl rollout undo
  • StatefulSet: stable pod identity and persistent storage; for databases
  • DaemonSet: one pod per node; for log collectors, monitoring agents
  • Service types: ClusterIP (internal), NodePort (external via node), LoadBalancer (cloud LB), ExternalName; headless service for StatefulSet DNS
  • ConfigMaps: non-sensitive config as env vars or volume-mounted files
  • Secrets: base64-encoded (not encrypted at rest by default; use Sealed Secrets or External Secrets Operator)
  • Persistent Volumes and PVCs; StorageClasses; dynamic provisioning
  • Essential kubectl: get, describe, logs -f, exec -it, apply -f, delete, rollout status, port-forward

Practice Deploy 3 microservices to minikube or kind; expose via Services; verify inter-service DNS resolution; test rolling update

Week 34 — Kubernetes Advanced: Ingress, HPA, Helm

  • Ingress: HTTP/HTTPS routing by host/path; TLS termination; NGINX Ingress Controller; annotations for rate limiting, auth
  • HPA (Horizontal Pod Autoscaler): scale based on CPU (default), memory, or custom metrics (Prometheus Adapter)
  • Probes: Liveness (restart if unhealthy), Readiness (remove from Service endpoints until ready), Startup (for slow-starting apps). Wrong config = traffic loss on deployment or restart loops
  • Resource requests and limits: requests = scheduler placement; limits = cgroup enforcement. OOMKill if memory limit exceeded. QoS classes: Guaranteed/Burstable/BestEffort
  • Rolling updates: maxSurge, maxUnavailable; blue-green (switch Service selector); canary (weight-based traffic split)
  • Helm: chart structure (templates, values.yaml, Chart.yaml); helm install/upgrade/rollback; templating ({{ .Values.x }}); Helm repositories
  • Namespace isolation; RBAC: Role, ClusterRole, RoleBinding, ClusterRoleBinding; service accounts

Practice Add HPA to Order service; configure readiness/liveness probes with Spring Boot actuator endpoints; create Helm chart for all 3 services; deploy with Helm

Week 35 — Observability + Service Mesh (Survey)

  • Grafana: dashboards from Prometheus; RED method panels (Rate, Errors, Duration); USE method; import community dashboards
  • Micrometer: metrics facade in Spring Boot; auto-instruments HTTP, JVM, HikariCP, Kafka; custom Counter, Timer, Gauge; exports to Prometheus via /actuator/prometheus
  • Distributed tracing: Micrometer Tracing (Spring Boot 3) with OTel bridge; traceparent header propagation; Jaeger or Zipkin backend; trace IDs in every log line
  • Structured logging: JSON logs (Logback + logstash-logback-encoder); correlation ID in MDC; Loki + Grafana or ELK for aggregation
  • Service mesh survey: Istio/Linkerd; sidecar proxy (Envoy); mTLS between services; traffic management (weighted routing, fault injection); observability without code changes

Week 37 — AWS Foundations: IAM, VPC, EC2, S3

  • IAM: users (long-lived — minimize), groups (apply policies to users), roles (assumed by services/EC2/Lambda — preferred), policies (JSON: Effect/Action/Resource); instance profiles; least privilege; sts:AssumeRole; service-linked roles

  • S3: buckets, objects (up to 5 TB), prefixes; storage classes (Standard, IA, Glacier); bucket policies vs ACLs; pre-signed URLs (temporary access); lifecycle rules; versioning; event notifications → Lambda/SQS; Cross-Origin Resource Sharing

Week 38 — Managed Services: RDS, EKS, Lambda, SQS/SNS

  • Lambda: function lifecycle (init → invoke → shutdown); cold start mitigation (Provisioned Concurrency, SnapStart for Java); concurrency limits; layers; destinations; Lambda + API Gateway (REST/HTTP API); Lambda + SQS/SNS/Kinesis/DynamoDB Streams triggers
  • SQS: standard (at-least-once, best effort ordering) vs FIFO (exactly-once, strict ordering); visibility timeout; dead-letter queues (DLQ); long polling; message filtering via attributes
  • SNS: pub/sub; topic subscriptions (SQS, Lambda, HTTP, email, SMS); message filtering; fan-out: SNS → multiple SQS queues

System Design + DSA + Interview Prep + Capstone

Month 11 System Design + DSA

Week 41–42 — System Design

  • Interview framework: (1) requirements (functional + non-functional: QPS, latency p99, availability %, data volume), (2) API design, (3) data model, (4) high-level architecture, (5) deep-dive bottlenecks, (6) failure modes + mitigations, (7) observability + cost
  • CAP theorem: under network partition, choose Consistency (reject requests) or Availability (return possibly stale data). Most systems choose AP + eventual consistency + explicit strong consistency where needed
  • Consistency models: strong, linearizable, sequential, causal, eventual, read-your-writes, monotonic read
  • Raft consensus: leader election, log replication, majority quorum, log compaction; split brain prevention via quorum requirement
  • Sharding: range (hot partitions risk) vs hash (even distribution, no range queries) vs directory-based; consistent hashing + virtual nodes; rebalancing strategies
  • Replication: synchronous (strong consistency, higher write latency) vs asynchronous (higher availability, risk of data loss); read replicas; leaderless (Dynamo-style, vector clocks)

Practice (one per day, 45 min + 15 min review): URL shortener · Rate limiter · Distributed cache · Notification system · News feed / timeline · Ride-sharing backend · Payment system · Video streaming service

Week 43–44 — Data Structures + Algorithms

  • Arrays/Strings: two pointers (sorted array pair sum, container with most water), sliding window (max sum subarray, longest unique substring), prefix sum (range sum query)
  • Linked lists: reverse, detect cycle (Floyd's), merge two sorted, find middle, remove nth from end
  • Stacks/Queues: monotonic stack (next greater element, largest rectangle), deque patterns (sliding window max), min stack
  • Trees: BST insert/search/delete, validate BST, BFS (level-order), DFS (preorder/inorder/postorder), lowest common ancestor, serialize/deserialize, diameter
  • Heaps: top-K elements (min-heap of size K), merge K sorted lists, kth largest element, median of data stream
  • Graphs: BFS (shortest path unweighted), DFS (connected components), Dijkstra (weighted shortest path), Union-Find (cycle detection, number of islands), topological sort (course schedule), Bellman-Ford
  • Dynamic programming: 0/1 knapsack, unbounded knapsack, LCS (longest common subsequence), LIS (longest increasing subsequence), coin change, matrix DP (unique paths, minimum path sum), interval DP
  • Sorting: merge sort (stable, O(n log n)), quicksort (avg O(n log n), in-place), counting sort / radix sort (linear for limited range); binary search variants (first/last occurrence, rotated array)
